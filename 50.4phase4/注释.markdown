3. 如果前端少传了字段，会报错吗？
------------------------------------------------------------
场景：Struct 有 A, B 两个字段，但前端只传了 {"a": "..."}。

- 默认行为 (Gin & 原生)：【不报错】。
  * 缺失的字段 B 会被赋予该类型的“零值”（字符串是 "", 整型是 0）。
  
- 如何让它报错？
  * 在 Struct 标签增加 `binding:"required"`。
  * 此时若缺了 B，ShouldBindJSON 会返回一个 non-nil 的 error。

- 如何区分“没传”和“传了空值”？
  * 使用指针：B *string `json:"b"`。
  * 若没传 B：变量 B 为 nil。
  * 若传了 {"b": ""}：变量 B 不为 nil，指向的内容是 ""。

4. ShouldBindJSON vs json.NewDecoder
------------------------------------------------------------
- ShouldBindJSON：Web 开发首选。一行代码搞定：`读取、解析、校验、报错`提示。
- NewDecoder：底层或非 Web 场景使用。纯粹的翻译官，`不带任何业务逻辑校验`。

空指针检查 (Nil Pointer)

防御措施： 在解引用指针、访问结构体字段或调用方法前，务必先进行 if p != nil 的判断。

切片越界防护 (Slice Out of Range)

防御措施： 访问特定索引前先检查 len(slice)；在遍历时，优先使用 for range 循环，它可以自动处理边界。

Map 安全初始化 (Map Initialization)

防御措施： 禁止直接向未初始化的 map (nil map) 写入数据。永远使用 m := make(map[KeyType]ValueType) 进行初始化。

安全类型断言 (Type Assertion)

防御措施： 避免使用单值断言 s := i.(string)，应使用双值安全形式：s, ok := i.(string)，并根据 ok 的值判断转换是否成功。

并发 Map 竞争保护 (Concurrent Map Access)

防御措施： 原生 Map 非线程安全。多协程读写时，需配合 sync.Mutex 或 sync.RWMutex 加锁，或者直接使用 sync.Map。

通道规范操作 (Channel Operations)

防御措施： * 严禁关闭 nil 通道。

严禁重复关闭通道。

遵循“由发送方负责关闭”的原则，防止向已关闭的通道发送数据。




# 为什么金融/计费系统（如 AI 网关）必须使用 int64 代替 float64

1. 浮点数精度陷阱 (The IEEE 754 Problem)
   - 计算机底层是二进制，无法完美表示 0.1, 0.2 等十进制小数。
   - 例子：0.1 + 0.2 在 Go 中会变成 0.30000000000000004。
   - 风险：在 AI 网关这种高频、微额扣费场景下，上百万次累加会导致严重的账务对不齐。

2. 整数的绝对精准性
   - int64 (整数) 在计算机运算中永远是精确的。1 + 2 永远等于 3。
   - 方案：将货币单位放大 N 倍（如 10,000 或 1,000,000 倍）。
   - 换算逻辑：
     - 若放大 100,000 倍：1 元 = 100,000 积分 (Amount)。
     - 充值 10 元 = 数据库存入 1,000,000。
     - 每次极小的 Token 消耗（如 0.00045 元）可以精准转换为整数 45 进行扣减。

3. 内部算力点 (Internal Credits) 的概念
   - AI 网关通常引入“算力点”的概念。
   - 这样做的好处：
     - 屏蔽真实货币汇率波动。
     - 数据库操作只有整数加减，性能更高。
     - 原生支持 WHERE balance >= cost 这种硬核行锁判定。

4. 架构师建议 (Best Practice)
   - 全链路（配置读取、逻辑运算、数据库存储、中间件校验）统一使用 int64。
   - 只有在“UI 展示层”即给用户看的时候，才将 int64 除以倍数转为 float 展示。
   - 确保系统在任何并发、任何规模下，账单永远一分不差。
